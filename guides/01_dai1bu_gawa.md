# 構築指示書 第1部「ガワ」— 作業環境をつくる

> **この指示書の上に書かれている「あなたの決定」を読んでから、以下の手順を実行すること。**
> 「あなたの決定」には会社名・担当者名・秘書名・選んだ役割・連携ツール・権限方針（6つのトグルのON/OFF）が書かれている。この第1部では、その決定内容を実際のフォルダ設定・ファイルに変換する。ここで作るのはAI秘書の「体（ガワ）」＝作業場所とルールブック。中身（会社知識）は第3部、能力（スキル）は第2部で追加する。

## 1. フォルダを作成する

プロジェクト直下に次の3つを作成する。

- `context/` — 会社の正しい情報を置く場所（ストック情報）。さらに用途別のサブフォルダを提案し、「あなたの決定」の役割・連携ツールに合わせて必要なものだけ作る。
  - `context/company/` — 会社概要・沿革・組織図
  - `context/customers/` — 顧客リスト・取引先情報
  - `context/products/` — 商品・サービス資料、料金表
  - `context/faq/` — よくある質問と回答、社内用語集
- `work/` — 日々の作業ファイルを置く場所（毎日入れ替わるフロー情報）
- `.claude/` — 設定・スキル・権限を格納する場所（このあと作るCLAUDE.mdとsettings.jsonの置き場）

既にこれらのフォルダがある場合は作り直さず、中身を確認したうえで流用する。

役割に合う分類が無い資料は、役割名のサブフォルダを作ってよい（例：経理・財務なら`context/keiri/`に勘定科目表・支払先一覧を置く）。company/customers/products/faqの4分類に無理に収めようとしない。

## 2. CLAUDE.md を作成する

`CLAUDE.md` はAI秘書が毎回のセッションで自動的に読み込む「取扱説明書」。長文にせず、次の構成で簡潔に書く。

### 2-1. 絶対ルール（最上部に固定で入れる）

1. 常に日本語で応答する
2. 社外・顧客向けの送信物（メール・資料・SNS投稿など）は、必ず下書きを提示し、担当者本人の許可を得てから送信する
3. 機密情報（顧客名・金額・個人情報など）は外部に送る文書・メッセージには一切含めない

### 2-2. 会社・本人・役割

「あなたの決定」に書かれた内容をそのまま転記する。

```markdown
## 会社・本人・役割
- 会社名：<会社名>
- 担当者（採用者）：<担当者名>
- 役割：<選んだ役割（複数可）> の業務を支援する秘書として、資料整理・連絡・スケジュール調整を行う
- 連携ツール：<連携ツール一覧>
```

### 2-3. AI秘書の名前と振る舞い

「あなたの決定」の秘書名を使い、一人称・口調を決めて明文化する（例：一人称は「わたし」、丁寧語、結論から簡潔に話す）。この節が、AI秘書の人格の唯一の設定源になる。

```markdown
## AI秘書について
- 名前：<秘書名>
- 一人称・口調：<決めた内容>
- 振る舞い：分からないことは推測せず担当者に確認する。判断が分かれる作業は必ず一言相談してから進める。
```

## 3. .claude/settings.json で権限を設定する

「あなたの決定」の権限方針は、6つのトグル（①外部送信の下書き確認／②自動コミット／③読み取り系コマンドの自動許可／④機密フォルダ書き込みの都度確認／⑤自動push／⑥Webアクセス許可）のON/OFFで示される。これを実際にツールの動きを制御する `permissions` に変換する。**破壊的コマンドと機密フォルダのdenyは、トグルの組み合わせによらず必ず固定で入れる。**

### 3-1. 固定deny（トグルの組み合わせによらず常に入れる）

```json
{
  "permissions": {
    "deny": [
      "Bash(rm -rf:*)",
      "Bash(git push --force:*)",
      "Bash(git reset --hard:*)",
      "Bash(format:*)",
      "PowerShell(Remove-Item -Recurse:*)",
      "Read(./secrets/**)",
      "Read(./.env)",
      "Edit(./secrets/**)",
      "Edit(./.env)"
    ]
  }
}
```

secrets/フォルダや.envを使わない場合は、その2行を省いてよい。

### 3-2. トグルごとの設定

| トグル | ONの場合 | OFFの場合 |
|---|---|---|
| ① 外部送信は下書き→許可制 | メール送信・Slack投稿など外部送信系ツールを `permissions.ask` に登録し、送信前に必ず確認を挟む | 都度確認なしで送信させる場合は、CLAUDE.mdにその旨を明記し、誤送信時の対応方針も決めておく |
| ② タスク完了で自動コミット | 作業がひと区切りつくたびに `git add` → `git commit` を自動で行う運用にする。コミットメッセージの書式をCLAUDE.mdに明記する | 明示的に指示があったときのみコミットする |
| ③ 読み取り系コマンドは自動許可 | `Bash(ls:*)` `Bash(grep:*)` `Bash(cat:*)` `Bash(find:*)` 等の読み取り系コマンドを `permissions.allow` に登録する | 読み取り系コマンドも都度確認する（allowに追加しない） |
| ④ 機密フォルダへの書き込みは毎回確認 | `secrets/` など機密フォルダへの `Write` / `Edit` を `permissions.ask` に登録する（読み取りは3-1の固定denyで別途保護済み） | 非推奨。機密フォルダへの誤書き込みが起きうるため、基本的にONのまま運用する |
| ⑤ コミット後に自動push | `git push` も自動で実行する運用にする | pushは明示的な指示があったときのみ実行する（askのままにするか未登録にする） |
| ⑥ Webアクセスを許可 | `WebFetch` `WebSearch` 等を `permissions.allow` に登録する | Web系ツールは `permissions.ask` のままにするか登録しない |

例：①②③④をON、⑤⑥をOFF（推奨の初期値）にした場合の完成形。

```json
{
  "permissions": {
    "allow": [
      "Bash(ls:*)",
      "Bash(grep:*)",
      "Bash(cat:*)",
      "Bash(find:*)"
    ],
    "ask": [
      "Bash(git push:*)",
      "Write(./secrets/**)",
      "Edit(./secrets/**)"
    ],
    "deny": [
      "Bash(rm -rf:*)",
      "Bash(git push --force:*)",
      "Bash(git reset --hard:*)",
      "PowerShell(Remove-Item -Recurse:*)",
      "Read(./secrets/**)",
      "Read(./.env)",
      "Edit(./secrets/**)",
      "Edit(./.env)"
    ]
  }
}
```

外部送信系のツール（Gmail・Slack等のMCPツール）は接続方式によって実際のツール名が変わるため、接続後の正式なツール名を確認して `permissions.ask` に追記する。既存の`.claude/settings.json`がある場合は上書きせず、上記のエントリを既存の配列に追記する。

## 4. 連携ツールを接続する

「あなたの決定」の連携ツール一覧を、**優先順位（コネクタ → CLI → 自作MCP）**で接続する。同じツールに複数の方法がある場合、上から順に試す。

1. まず公式コネクタ（MCP）が使えないか確認する。管理者権限が必要なものは担当者に依頼する
2. コネクタが無ければ、そのツールの公式CLIがないか確認する
3. どちらも無ければ、最小権限の自作MCP（API直叩き）を検討する。この場合はAPIキーの置き場（後述）を先に決める

| ツール | 接続方法 |
|---|---|
| Slack | 公式Slackコネクタ（MCP）を最優先で接続し、投稿先チャンネルと権限範囲を確認する |
| Gmail | 公式Gmailコネクタ（MCP）を接続し、秘書用に権限範囲（送信・下書きのみ等）を絞る |
| Googleカレンダー | 公式Googleカレンダーコネクタ（MCP）を接続し、閲覧・作成権限を確認する |
| Google Drive | 公式Google Driveコネクタ（MCP）を接続し、アクセスできる共有ドライブの範囲を確認する |
| Notion | 公式Notionコネクタ（MCP）があれば接続。無ければNotion CLIか自作MCPを検討する |
| Chatwork | 公式コネクタが無いことが多いため、CLIか自作MCP（Chatwork API）での接続を検討する |

## 5. 動作確認

すべて設定できたら、担当者に対して「おはよう」と話しかけてもらい、次の2点が返るか確認する。

- AI秘書としての簡単な自己紹介（名前・支援する役割）
- 今日の段取り（分かる範囲でよい。連携ツール未接続なら「まだ予定を確認できません」と正直に答える）

意図した振る舞いと違えば、CLAUDE.mdの記載を見直す。

## 付録：セキュリティ最低限

- APIキー・パスワードは `secrets/` フォルダか `.env` のどちらかに置き、CLAUDE.mdやSKILL.mdなど読み込まれるドキュメントには直書きしない
- 上記3-1の固定denyは、Read/Editツール経由の読み取りしかブロックできない。Bash/PowerShell経由での読み取りまでは防げないため「絶対に読めない」ではなく事故防止の第一層と理解する
- gitを使う場合は`secrets/`と`.env`を`.gitignore`にも追加する
