# 構築指示書 第2部「能力」— スキルとチームをつくる

> **この指示書の上に書かれている「あなたの決定」（受講する研修＝スキル一覧）を読んでから、以下の手順を実行すること。**
> 第1部でAI秘書の体（フォルダ・ルール）ができた。第2部では、実際に手を動かせる「能力」＝スキル（スラッシュコマンド）とAgent Team（複数エージェントの分担）を組み立てる。

## 1. スキルの作り方（共通ルール）

スキルは `.claude/skills/<スキル名>/SKILL.md` という1ファイルで定義する。1スキル＝1フォルダ。

```markdown
---
name: <スキル名（英数字とハイフン）>
description: <このスキルを「いつ使うか」を具体的に書く。ここが自動起動の判定材料になるため、トリガーになる言葉（「〜して」「〜まとめて」等）を含める>
---

# <スキル名>

## 目的
<何のためのスキルか1-2行>

## 入力
<何を渡すか（例：カレンダーの予定、メール文面、レシート画像）>

## 手順
1. ...
2. ...

## 出力
<何を返すか（例：段取り表、チェック結果、起票データ）>

## Gotchas（過去の失敗メモ）
<運用中に見つかった落とし穴をここに追記していく。作成時点は空欄でよい>
```

**descriptionの書き方が重要**：「◯◯して」「◯◯まとめて」のようにユーザーが実際に話しかける言い回しを具体的に書くと、そのスキルが自動的に起動する。抽象的な説明だけだと起動しない。

**Gotchas枠は必ず作る**：一度失敗した・想定外だった挙動を書き足していく場所。次に同じスキルを使うときや、似たスキルを新しく作るときの事故防止になる。

## 2. universalスキル（全員に必ず作る2つ）

配属先や選んだ研修に関わらず、以下の2つは最初に作成する。

### 2-1. `/daily-schedule`（朝の段取り表）

- **目的**：カレンダーとタスクから、今日の段取り表を作る
- **入力**：接続済みのカレンダー（Googleカレンダー等）の今日の予定、work/内の未完了タスクメモ
- **手順**：
  1. 今日の予定を時系列に並べる
  2. 予定の合間に処理できそうなタスクを差し込む
  3. 優先度が高い・締切が近いタスクを冒頭に出す
- **出力**：時刻順の段取り表（予定＋差し込みタスク）。担当者が朝一番に見て動けるように、要確認事項があれば先頭に出す
- **Gotchas**：（運用開始後に追記）

### 2-2. `/mtg-prep`（次の会議の事前情報まとめ）

- **目的**：次に控えている会議の参加者・議題・過去の関連メモをまとめて事前に渡す
- **入力**：カレンダーの次の予定（会議）、context/customers内の関連情報、過去の議事録（あれば）
- **手順**：
  1. 会議の参加者・時間・場所（オンライン/対面）を特定する
  2. 参加者に関する情報をcontext/から拾う（初対面か、過去のやり取りがあるか）
  3. 議題が分かれば、話すべき論点・確認したい数字を整理する
- **出力**：会議名・参加者・想定論点・持参すべき資料のミニまとめ
- **Gotchas**：（運用開始後に追記）

## 3. 役割別スキル（「あなたの決定」の研修一覧から作る）

「あなたの決定」に記載された研修（スキルID・コマンド名・概要）を1つずつ、次の手順で具体化する。**概要文だけでは実装に足りないため、必ず担当者に会話でヒアリングしてから作る。**

1. 研修一覧から1件選ぶ
2. 担当者に次を聞く（AskUserQuestion等の選択式が使えるならそれで聞く）
   - 具体的にどんな入力を渡すことが多いか（例：レシート画像／メールの下書き／会議メモ）
   - 出力はどんな形式が使いやすいか（表・箇条書き・特定のシート等）
   - 過去に困った・失敗した経験があるか（あればGotchasの初期エントリにする）
3. ヒアリング内容をもとに、上記フォーマットでSKILL.mdを作成する
4. 概要文にある名称・コマンド名は「あなたの決定」の表記のまま使う（担当者が覚えている名前を変えない）
5. 全件作り終えたら一覧で「修了証：発行済み」として報告する

## 4. Agent Team（複数エージェントの分担）

1つのスキルで完結しない、複数工程にまたがる仕事は、サブエージェントに分担させる「Agent Team」を組む。定義は `.claude/agents/<エージェント名>.md` に1体ずつ置く。

### 組み方

1. 仕事を工程に分解する（例：顧客リサーチ→競合分析→設計→執筆→レビュー）
2. 工程ごとに担当エージェントを1体ずつ定義する。各エージェントのfrontmatterに`description`（いつ呼ばれるか）と、必要なツールだけを絞って書く
3. 最後の工程（レビュー等）は、前工程の担当とは別のエージェントにする（自分の作業を自分で採点しないため）
4. 呼び出し順（フロー）をメインのCLAUDE.mdかチーム定義の冒頭に明記する

### 例：顧客リサーチ→競合分析→設計→執筆→レビュー

```
1. researcher   : 顧客リサーチ（対象企業の情報収集）
2. competitor   : 競合分析（同業他社との比較）
3. planner      : 構成設計（1・2を踏まえた提案の骨子）
4. writer       : 執筆（骨子から本文作成）
5. reviewer     : レビュー（誤り・トーン・機密混入チェック）
```

**まずは単発スキルから慣れることを推奨する。** Agent Teamは工程間の受け渡しミスが起きやすいため、単発スキル（第2部の2〜3節）で運用に慣れてから、複数工程にまたがる仕事が増えてきたタイミングで組むのがよい。

## 5. 動作確認

作成したスキルのうち1つを実際に呼び出してみる（例：`/daily-schedule`）。想定した入力を渡し、出力が仕様どおりの形式で返るか確認する。ズレがあればSKILL.mdの手順を修正する。
